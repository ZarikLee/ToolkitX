## Context

ToolkitX 需要从"运维管理平台"重新定位为"运维工具箱 + AI 助手"。核心转变：
- 不再需要客户权限（合规）
- 每个功能解决真实痛点（实用）
- 引导式设计 + AI 辅助（易用）

## Goals / Non-Goals

**Goals:**
- 配置生成器：引导式填写 → 生成配置文件
- 脚本库：Python/SQL/Shell 脚本模板 + AI 生成
- 诊断工具：从服务器视角探测网络问题
- 数据处理：JSON/YAML/编码/时间戳等常用转换
- 在线终端：差异化（命令库、笔记本、会话快照）

**Non-Goals:**
- 不做容器管理（不需要 Docker API）
- 不做数据库管理（不需要连接客户数据库）
- 不做服务器监控（不需要 agent）
- 不做多用户权限系统

## Decisions

### 配置生成器：模板 + 引导式
- **选择**: 预置模板 + 分步填写 + 每项说明
- **理由**: 实施工程师不一定每种配置都会写
- **AI 辅助**: 用户描述需求 → AI 生成配置

### 诊断工具：服务端执行
- **选择**: API 调用 → 服务器执行 → 返回结果
- **理由**: 浏览器无法做 TCP/ICMP 探测
- **实现**: child_process 执行 ping/nc/dig 等命令

### 脚本库：模板 + AI 生成
- **选择**: 预置常用脚本模板 + AI 按描述生成
- **理由**: 实施工程师需要但不一定写得出
- **格式**: 在线编辑 + 一键复制/下载

### 终端差异化：运维工作台
- **选择**: 不跟 xShell 比 SSH 功能，比工作流
- **差异点**: 命令库、命令笔记本、会话快照、与诊断联动

### 数据存储：JSON + localStorage
- **选择**: 配置存 JSON，用户偏好存 localStorage
- **理由**: 简单，无需数据库

## Risks / Trade-offs

- **[AI 依赖]** 需要 AI API → 先做模板，AI 后续迭代
- **[诊断能力]** 服务器没装工具 → 提供安装脚本
- **[性能]** 多个工具并行 → 按需加载，不阻塞
